==================================================
FINAL ADDITIONAL PRODUCT RULES
==============================

==================================================
COMPLETED TASKS SECTION
=======================

Completed tasks should remain visible on the dashboard.

Instead of mixing completed and incomplete tasks together, create a separate section:

Today's Targets

☐ Workout

☐ Read OS

---

Completed Today

☑ DSA Problems

☑ System Design Notes

This provides a satisfying sense of accomplishment while keeping active tasks clear.

==================================================
FRIEND REMOVAL
==============

Users can remove friends.

Example:

Connected Friends

🚀 Rahul

[Remove Friend]

Removing a friend:

* Removes visibility into each other's activity.
* Removes them from friend leaderboards.
* Removes them from connected friends lists.

However:

Users may send friend requests again in the future.

Removing a friend does NOT permanently block future friendship.

==================================================
NOTIFICATION STORAGE
====================

Notification retention should be implemented using whichever approach best fits the existing database architecture and performance requirements.

Priority:

1. Simplicity
2. Maintainability
3. Performance

Notifications may be retained indefinitely or archived according to implementation needs.

Choose the approach that best suits the codebase.

==================================================
EMAIL SEARCH BEHAVIOR
=====================

Friend requests are sent using email addresses.

If the email does not belong to an existing account:

Display:

"No user found with this email."

Do not create placeholder users.

Do not create pending invitations for non-existent accounts.

==================================================
PROFILE SETTINGS PAGE
=====================

Create a dedicated Profile Settings page.

Users can update:

* Display Name
* Profile Emoji

Users cannot update:

* Authentication Email

Profile Settings should be easily accessible from navigation or user menu.

==================================================
TIMEZONE SUPPORT
================

All day-based calculations must use the user's local timezone.

This includes:

* Daily Targets
* Daily Score
* Calendar Days
* Streaks
* Carry-over Logic

Avoid server-time based calculations that could break streaks incorrectly.

==================================================
MIDNIGHT PROCESSING
===================

At the start of a new day:

* Carry over unfinished targets.
* Recalculate streak state.
* Start a new daily score record.

This should happen automatically.

==================================================
TASK DELETION RULES
===================

Users may delete tasks.

Example:

3 tasks exist

☑ DSA
☑ Workout
☐ Reading

Delete:

☐ Reading

Result:

2 tasks remain

☑ DSA
☑ Workout

Completion becomes:

2 / 2

100%

Streak calculations should use the final state of the day's tasks.

Deleting an unfinished task may therefore affect completion percentage and streak outcomes.

==================================================
IMPLEMENTATION PRIORITY
=======================

Do not spend additional time searching for more design inspiration.

Use the existing references already provided.

Focus implementation effort on:

* Dashboard polish
* Task interactions
* Leaderboard experience
* Friend activity visibility
* Calendar experience

Prioritize execution over collecting more references.
